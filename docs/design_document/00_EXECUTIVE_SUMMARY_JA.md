# 00. Executive Summary: geo-line-ranker とは何か

## 一言でいうと

`geo-line-ranker` は、**地理条件・路線条件を主軸にして、学校・イベント・記事などの候補をルールベースで並べる推薦基盤**です。

AI や生成モデルで「それっぽくおすすめ」を作るのではなく、次のような明確なルールで順位を決めます。

- ユーザーが探しているエリアに近いか
- ユーザーが見た学校や保存した学校に近いか
- 同じ路線・近い駅にあるか
- イベントが新しいか、重要度が高いか
- 同じ学校や同じグループばかり出ていないか
- データが足りない場合でも、安全なフォールバックになっているか

## なぜ作るのか

学校・イベント探しでは、ユーザーにとって「近い」「通いやすい」「関心に合っている」が非常に重要です。

一方で、一般的な推薦システムは次の問題を抱えやすいです。

- なぜ表示されたか説明しにくい
- 遠すぎる候補を出してしまう
- 新規ユーザーに弱い
- AI / ML 前提になり、データ量や運用コストが大きい
- 個人情報の扱いが複雑になりやすい

`geo-line-ranker` は、これらを避けるために **決定論的なルールベース推薦**として設計します。  
決定論的とは、同じ入力なら同じ結果が返るという意味です。運用・検証・説明がしやすくなります。

## 何をおすすめできるか

初期の想定は次の3種類です。

| 種類 | 例 | 状態 |
|---|---|---|
| 学校 | 学校詳細ページ | 実装対象 |
| イベント | 説明会、オープンキャンパス、相談会 | 実装対象 |
| 記事 | お知らせ、読み物、特集 | 将来拡張枠 |

現在の実装では、`school` と `event` を中心に扱い、`article` は将来のための予約枠として扱います。

## 使う人

| 利用者 | 期待すること |
|---|---|
| 企画担当 | どんな推薦ができるかを理解する |
| PM | 要件と実装のズレを確認する |
| エンジニア | API、DB、worker、crawler を実装・運用する |
| データ担当 | 学校・駅・イベントデータを投入する |
| 法務・セキュリティ | 個人情報やクローリング方針を確認する |
| OSS利用者 | ローカルで動かして仕組みを試す |

## システム全体のざっくり図

```text
ユーザーの条件・行動
  例: エリア、路線、閲覧、保存、イベントクリック
        ↓
API
  例: POST /v1/recommendations
        ↓
候補を探す
  PostgreSQL/PostGIS または OpenSearch
        ↓
Rust のランキング処理
  地理・路線・行動・人気・多様性でスコアリング
        ↓
推薦結果
  学校・イベント + 推薦理由 + スコア内訳
```

## このOSSがやらないこと

このプロジェクトは、あえて次をやりません。

- AI / 生成AIによる推薦文生成
- 機械学習モデルの学習・推論
- 端末GPSを前提にしたリアルタイム位置推薦
- 無制限のWebクローリング
- Redis を正本にしたデータ管理
- OpenSearch だけにすべてを任せる推薦

できることを増やすより、**壊れにくく、説明でき、検証できること**を優先します。

## 成功の基準

この基盤が成功している状態は、次のように判断します。

- 推薦枠からのクリック率が上がる
- 学校詳細・イベント詳細への回遊が増える
- 遠すぎる候補が出にくい
- 推薦理由を人間が説明できる
- 問題が起きたとき、入力・候補・スコアを追跡できる
- ローカル環境で誰でも再現できる

## 非エンジニア向けの理解ポイント

一番大切なのは、`geo-line-ranker` は「魔法のおすすめAI」ではないということです。

これは、透明な定規のようなものです。  
地理、路線、行動、人気、鮮度という複数の定規で候補を測り、近くて関係があり、偏りすぎない順番に並べます。

そのため、導入時に大切なのは「どの定規を強くするか」を決めることです。  
たとえば、検索結果ページでは学校を強めに、トップページではイベントを少し強めにする、といった調整ができます。
