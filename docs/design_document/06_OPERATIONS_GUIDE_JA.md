# 06. 非エンジニア向け運用ガイド

## このドキュメントの目的

この文書は、開発者ではない人が `geo-line-ranker` の状態を確認し、運用上の判断をするためのガイドです。

ここでは細かいコマンドの意味よりも、**何を見れば安全か**、**何が起きたら危ないか**、**誰に何を確認すべきか**を整理します。

## 日々見るべきもの

| 見るもの | 目的 |
|---|---|
| APIが起動しているか | 推薦が返るか |
| Workerが動いているか | 行動ログや同期が処理されているか |
| Import / Crawl の結果 | データ取り込みが成功したか |
| Cache の状態 | 古い推薦が残っていないか |
| 推薦結果のサンプル | 遠すぎる候補や偏りがないか |
| エラー率 | 実装・データ・外部ソースの異常検知 |

## 推薦結果のチェック観点

### 1. 遠すぎる候補が出ていないか

最重要です。

例:

- 東京の駅を指定したのに北海道や沖縄が上位に出る
- 神奈川県の検索で、関係の薄い関西の候補が上位に出る

この場合は、候補抽出・fallback・距離ペナルティの確認が必要です。

### 2. 同じ学校ばかり出ていないか

同じ学校のイベントが上位に並びすぎると、探索体験が狭くなります。

見るポイント:

- 同一学校の件数
- 同一グループの件数
- 学校とイベントの比率

### 3. 推薦理由が正しいか

推薦理由が嘘になるのは危険です。

例:

- 「田町駅に近い」と出ているが実際は遠い
- 「最近見た学校に近い」とあるが履歴がない
- 「同一路線」とあるが路線が違う

この場合は、score breakdown や説明文生成ルールを確認します。

### 4. フォールバックしすぎていないか

常に `national_safe` に落ちている場合、データ不足や候補抽出の問題があります。

確認すること:

- 学校・駅リンクがあるか
- イベントがactiveか
- placement profile が厳しすぎないか
- OpenSearch projection が同期されているか

## データ取り込み時の確認

CSV import や crawl 後は、次を確認します。

- 取り込み件数
- スキップ件数
- 重複件数
- エラー内容
- source id
- checksum
- 実際に推薦結果に反映されたか

## 事故になりやすいケース

### CSVの列名が変わった

症状:

- import 失敗
- 取り込み件数が0
- イベント名や日付が空になる

対処:

- CSV schemaを確認
- staging結果を見る
- 変更が継続的ならimport仕様を更新

### WebページのHTMLが変わった

症状:

- parse 失敗
- 以前は取れていたイベントが消える
- parser health が悪化

対処:

- fetch は成功しているか確認
- raw HTML を確認
- parser の expected shape を更新
- live_ready を一時停止する

### Redisに古い結果が残る

症状:

- データを更新したのに推薦結果が変わらない

対処:

- cache invalidation job を確認
- Redis を一時的に無効化して比較
- profile version / algorithm version が cache key に入っているか確認

### OpenSearchとDBがズレる

症状:

- full mode だけ推薦結果がおかしい
- sql_only では出る候補が full では出ない

対処:

- projection sync
- index rebuild
- PostgreSQL側の正本を確認

## 変更時のチェックリスト

ランキング設定を変えたら、最低限これを確認します。

- [ ] home / search / detail / mypage の各placementで確認した
- [ ] 遠すぎる候補が上位に出ていない
- [ ] 同じ学校ばかりになっていない
- [ ] 学校・イベントの比率が崩れていない
- [ ] 推薦理由が正しい
- [ ] fallback stage が想定内
- [ ] algorithm version / profile version を更新した
- [ ] テストデータで再現できる

## 誰に相談するか

| 問題 | 相談先 |
|---|---|
| 推薦順位がおかしい | 開発・PM |
| データが入らない | データ担当・開発 |
| クローリング可否 | 法務・セキュリティ・開発 |
| 個人情報の扱い | 法務・セキュリティ |
| UI表示がおかしい | フロントエンド |
| APIが落ちている | バックエンド・インフラ |

## 運用で守ること

- データ元が不明なファイルを入れない
- 権利が不明なWebページをcrawlしない
- いきなり本番で新しいparserを動かさない
- profile変更は記録する
- 推薦理由が説明できない設定にしない
- 問題が起きたら、まず sql_only minimal mode で比較する

## まとめ

運用で大切なのは、完璧な順位を一発で作ることではありません。  
「おかしい」と思ったときに、なぜおかしいかを追えることです。

`geo-line-ranker` は、そのために監査ログ、score breakdown、fallback stage、version を持ちます。
