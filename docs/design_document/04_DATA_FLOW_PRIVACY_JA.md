# 04. データフローとプライバシー

## 基本方針

`geo-line-ranker` は、推薦に必要なデータだけを扱います。  
特に住所や行動ログは、便利だからといって無制限に保存しません。

原則は次の通りです。

- 個人情報は最小限にする
- 推薦に不要な細かさでは持たない
- 正本は PostgreSQL に置く
- キャッシュは消えても問題ない設計にする
- 取り込み元とライセンスを記録する
- 監査ログを残す

## 扱うデータ

### 1. コンテンツデータ

| データ | 例 |
|---|---|
| 学校 | 学校名、所在地、学校コード、緯度経度 |
| イベント | イベント名、開催日、対象学校、申込URL |
| 駅・路線 | 駅名、路線名、座標、駅コード |
| 記事 | 将来拡張枠 |

### 2. ユーザー行動データ

| 行動 | 用途 |
|---|---|
| 学校詳細閲覧 | 関心の推定 |
| 保存・しおり | 強い関心 |
| 検索実行 | エリア・路線条件の理解 |
| イベント閲覧 | イベント関心 |
| 申込導線クリック | 強い関心 |
| 共有 | 関心の補助シグナル |

### 3. 地理・住所関連データ

| データ | 推奨する扱い |
|---|---|
| 都道府県 | 推薦文脈として利用しやすい |
| 市区町村 | 推薦文脈として利用可能 |
| 詳細住所 | 原則として直接推薦に使い続けない |
| 緯度経度 | 学校・駅には必要。個人には慎重 |
| 端末GPS | 初期スコープでは使わない |

## データの流れ

```text
公式データ / CSV / 許可されたcrawl
        ↓
一時保存 .storage/raw
        ↓
正規化・検証
        ↓
PostgreSQL/PostGIS
        ↓
API / Worker / Ranking
        ↓
Redis cache または API response
```

## 行動ログの流れ

```text
ユーザー操作
  例: 学校を見る、保存する、イベントを見る
        ↓
POST /v1/track
        ↓
user_events に保存
        ↓
worker が snapshot 更新ジョブを処理
        ↓
recommendation API のスコアリング材料になる
```

## 住所情報の扱い

住所情報は、推薦精度を上げられる一方で、扱いを間違えるとリスクになります。

推奨方針:

- 生の詳細住所を長期保存しない
- 都道府県・市区町村・近傍駅などの粗い特徴量へ変換する
- 利用目的を明示する
- 保持期間を決める
- 同意や利用可否をアプリ側で管理する
- 削除要求に対応できるようにする

## 端末位置情報について

初期設計では、端末GPSは利用前提にしません。

理由:

- 同意設計が重くなる
- リアルタイム性が必要になる
- 実装・運用・説明が複雑になる
- 検索条件や居住地情報だけでも十分に始められる

将来アプリ化した場合のみ、別仕様として検討します。

## キャッシュの扱い

Redis は API 応答を速くするためのキャッシュです。

守ること:

- Redis にしかない重要データを作らない
- Redis が消えても PostgreSQL から再計算できる
- ranking profile version や algorithm version を cache key に含める
- データ更新時はキャッシュを無効化する

## インポートと監査ログ

CSV import や crawl では、次を記録します。

- いつ取り込んだか
- どのファイル・URLから取り込んだか
- checksum
- 成功・失敗
- 何件入ったか
- 何件スキップしたか
- parser のエラー
- 重複排除の結果

これにより、問題が起きたときに「どの入力から壊れたか」を追えます。

## ライセンス管理

OSSでは、データライセンスが非常に重要です。

データソースごとに次を記録します。

- ソース名
- 配布元
- ライセンス
- 利用条件
- 取得日
- 再配布可否
- attribution の要否

特にオープンデータは、同じ分野でも年度や配布元により条件が違うことがあります。  
「前に使えたから今回も使える」と決めつけないことが重要です。

## 推薦トレース

推薦APIは、必要に応じて次を残せるようにします。

- request context
- candidate count
- fallback stage
- score breakdown
- explanation
- algorithm version
- profile version
- clicked / not clicked

これは、推薦品質の改善だけでなく、問い合わせ対応にも役立ちます。

## まとめ

このプロジェクトでは、データを「たくさん集める」より、**どこから来て、なぜ使い、いつ消すかを説明できること**を重視します。

推薦基盤はデータの川です。  
濁った川に立派な水車を置いても、よく回りません。まず水源をきれいにします。
