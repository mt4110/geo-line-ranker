# 01. 要件定義: 何を満たすべきか

## 目的

ユーザーの行動データと検索条件を使い、学校・イベントなどのコンテンツを「おすすめ」として表示します。  
目的は、ユーザーが目的の情報に早くたどり着き、サイト内を自然に回遊できるようにすることです。

ただし、この推薦は AI / 生成モデルでは行いません。  
行動や地理条件をスコア化し、ルールに基づいて順位を決めます。

## 基本方針

### 1. 非AI推薦であること

推薦は、機械学習モデルの学習や推論を前提にしません。

理由は以下です。

- 説明しやすい
- 小さいデータでも始められる
- 運用者がルールを検証しやすい
- 個人情報や同意設計を複雑にしすぎない
- OSSとしてローカルで再現しやすい

### 2. 地理・路線を必ず主軸にすること

ユーザーが北海道周辺を探しているのに、沖縄の候補を優先表示するような推薦は避けます。

候補抽出の主軸は次のいずれかです。

- 検索条件の都道府県・市区町村
- 検索条件の路線・駅
- アカウント情報から得られる居住地関連情報
- 直近の検索・閲覧行動から推定できるエリア

### 3. 推薦理由を出せること

ユーザーや運用者が「なぜこれが出たのか」を理解できるようにします。

例:

- 「田町駅に近い候補です」
- 「最近見た学校と同じ路線にあります」
- 「このエリアで人気のイベントです」
- 「保存した学校に近い学校です」

この理由はAIで文章生成しません。  
ルールに対応した固定文言を組み合わせます。

### 4. データ不足でも壊れないこと

新規ユーザーや検索条件がないユーザーでも、推薦枠を空にしない設計にします。

フォールバックの順序は、基本的に次のようにします。

1. 検索条件のエリア・路線
2. アカウントの居住地関連情報
3. 直近行動から得られるエリア・路線
4. 同一都道府県または近隣エリアの人気順
5. 全国人気。ただし極端に遠い候補は抑制

### 5. 偏りを抑えること

同じ学校ばかり、同じ系列ばかり、イベントばかりが出る状態を避けます。

たとえば、上位10件の中に同じ学校のイベントが5件並ぶと、ユーザーにとって探索の幅が狭くなります。  
そのため、次のような上限を設けます。

- 同一学校の最大表示数
- 同一学校グループの最大表示数
- 学校・イベントなどコンテンツ種別ごとの最大比率

## 推薦対象

| 対象 | 説明 | 初期状態 |
|---|---|---|
| 学校 | 学校詳細ページへの誘導 | 対象 |
| イベント | 学校イベント、説明会、オープンキャンパス等 | 対象 |
| 記事 | お知らせ、特集、コラム等 | 将来拡張 |

## 入力データ

### エリア・路線情報

推薦の主軸です。候補を絞るために必ず使います。

| 入力 | 例 |
|---|---|
| 都道府県 | 東京都、神奈川県 |
| 市区町村 | 港区、八王子市 |
| 駅 | 田町駅、品川駅 |
| 路線 | 山手線、京浜東北線 |
| 居住地 | アカウント情報から取得できる範囲 |

### 行動イベント

ランキングの材料です。

| 行動 | 意味 |
|---|---|
| 学校詳細閲覧 | 関心の弱〜中シグナル |
| 保存・しおり | 関心の強シグナル |
| 検索実行 | 条件の明示シグナル |
| イベント詳細閲覧 | イベント関心 |
| 申込導線クリック | 強い関心 |
| 共有 | 関心・他者共有 |
| 直近閲覧 | リセンシー、最近の関心 |

## 出力

API は推薦結果を返します。

返す内容は次の通りです。

- 推薦アイテム一覧
- 学校またはイベントのID
- 表示名
- 駅・路線などの補足情報
- スコア
- 推薦理由
- どのフォールバック段階だったか
- 使用したランキングルールのバージョン

## 非目標

このプロジェクトでは、次は初期スコープに入れません。

- AIによる推薦
- 生成AIによる推薦理由の文章生成
- GPS前提の位置情報推薦
- 個人情報を細かく長期保持する仕組み
- 無許可・広範囲のスクレイピング
- すべてのデータソースを自動収集する仕組み

## 要件の見直しタイミング

以下のときは、要件を見直します。

- 推薦対象に記事など新しい種類を追加する
- 住所情報の利用範囲が変わる
- クローリング対象を増やす
- OpenSearch を本番利用する
- 行動イベントの重みを変える
- 法務・セキュリティ上の制約が変わる
